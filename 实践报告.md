# ToyC 编译器实践报告

## 1. 项目概述

本项目实现了一个面向 `ToyC` 语言的编译器。编译器从标准输入流读取 `ToyC` 源程序，经过词法分析、语法分析、语义分析和代码生成后，输出 `RISC-V32` 汇编代码。

当前实现的目标优先级是功能正确性，在此基础上尽可能保证项目结构清晰、测试可复现。性能优化尚未作为当前阶段重点。

## 2. 开发环境

- 实现语言：`C++20`
- 构建方式：`Makefile`、`CMakeLists.txt`
- 本地回归测试：`make test`

## 3. 整体架构

项目采用经典编译器流水线设计：

1. `Lexer`：将输入字符流切分为 token 序列。
2. `Parser`：将 token 序列解析为抽象语法树（AST）。
3. `SemanticAnalyzer`：检查作用域、常量约束、函数调用规则等语义条件。
4. `CodeGenerator`：将 AST 翻译为 `RISC-V32` 汇编。
5. `riscv_runner.py`：对生成的汇编做轻量级解释执行，用于本地端到端验证返回值。

## 4. 各模块设计

### 4.1 词法分析

词法分析器支持以下 token：

- 关键字：`const`、`int`、`void`、`if`、`else`、`while`、`break`、`continue`、`return`
- 标识符
- 整数字面量
- 运算符与界符：`+ - * / % ! && || < > <= >= == != = ( ) { } , ;`
- 注释：`//` 单行注释与 `/* ... */` 多行注释

为贴合题目中 `NUMBER` 的定义，词法分析支持负整数字面量，并正确处理 `-2147483648` 这样的边界情况。

### 4.2 语法分析

语法分析器采用递归下降实现，覆盖了课程文档中给出的核心语法：

- 全局变量声明、全局常量声明
- `int` / `void` 函数定义
- 语句块、空语句、表达式语句、赋值语句
- `if-else`、`while`、`break`、`continue`、`return`
- 算术、关系、逻辑、一元表达式
- 函数调用与参数列表

分析结果会构造成 AST，便于后续语义检查和代码生成。

### 4.3 语义分析

当前语义分析覆盖了以下规则：

- 程序必须定义 `int main()`
- 全局名称不能重复定义
- 局部变量、局部常量、参数不能在同一作用域重复声明
- 标识符必须先声明后使用
- `const` 不能作为赋值语句左值
- `const` 初始化表达式必须在编译期可确定
- `break` / `continue` 只能出现在循环体内
- `void` 函数不能返回值
- `int` 函数必须在所有控制路径上返回值
- 函数调用必须出现在被调函数定义之后，但允许函数内部递归调用自身
- `void` 函数调用不能出现在需要值的位置
- 如果某个变量遮蔽了函数名，则该标识符不能再被当作函数调用

### 4.4 代码生成

后端当前采用直接从 AST 到 `RISC-V32` 汇编的方式，不经过额外 IR。实现了如下能力：

- 局部变量和参数的栈帧布局
- 全局变量/全局常量的 `.data` 段生成
- 编译期可确定的全局初始化
- 运行期全局初始化函数 `__toyc_init_globals`
- 算术、比较、逻辑表达式生成
- `if-else`、`while`、`break`、`continue`
- 函数调用、递归调用、8 个以上参数的调用
- 基本的调用栈对齐处理

当前实现的代码生成重点是正确性而不是优化，因此寄存器使用策略仍然比较朴素，主要依赖栈临时保存中间值。

## 5. 测试方案

项目内置了回归测试，执行方式如下：

```bash
make test
```

测试分为两类：

1. `tests/pass`：合法 `ToyC` 程序，要求编译成功；其中部分样例还会通过 `riscv_runner.py` 解释执行生成的汇编，并检查最终返回值。
2. `tests/fail`：非法程序，要求在词法/语法/语义阶段给出预期错误。

当前回归样例覆盖了：

- 基础表达式、负数字面量、一元运算
- 注释
- 局部变量、全局变量、全局常量
- 运行期全局初始化与依赖链
- `if-else`、`while`、`break`、`continue`
- 短路求值
- 递归
- 作用域遮蔽
- 嵌套函数调用
- 多参数函数调用
- 典型语义错误（重复定义、参数不匹配、非法 `main`、`void` 误用等）

## 6. 当前完成情况

目前已经完成了一个功能较完整的 `ToyC -> RISC-V32` 编译器雏形，并且具备较系统的自动回归测试能力。

已完成部分：

- 词法分析
- 语法分析
- AST 构建
- 主要语义检查
- 基础代码生成
- 本地端到端运行验证

尚可继续完善的部分：

- 更多边界语义样例
- 更复杂程序的运行验证
- 后端寄存器分配优化
- 生成代码性能优化

## 7. 遇到的问题与解决思路

### 7.1 没有现成的 RISC-V 运行环境

本地环境缺少直接运行 `RISC-V32` 程序的工具链。为了解决“只能看汇编、不能验结果”的问题，项目内实现了一个轻量级解释执行器 `tests/riscv_runner.py`，用于对当前生成的汇编子集做端到端返回值验证。

### 7.2 运行期全局初始化

部分全局变量初始化表达式可能依赖函数调用或先前的全局变量，因此不能简单在 `.data` 段中全部静态展开。为此实现了专门的初始化函数，并在 `main` 开始时执行一次初始化逻辑。

### 7.3 函数调用栈对齐与嵌套调用

在朴素代码生成中，嵌套函数调用和超过 8 个参数的调用容易破坏栈平衡。当前实现通过显式计算调用前的栈填充量，尽量保证调用过程中的栈布局稳定。

## 8. 总结

本项目已经实现了 `ToyC` 语言核心子集的完整编译流程，并建立了较扎实的自动测试基础。虽然当前版本仍以正确性优先、尚未深入优化，但已经具备继续扩展和完善的良好结构。
